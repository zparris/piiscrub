# Releasing PIIScrub

This document explains how to publish a new version of PIIScrub as a downloadable desktop app for non-technical users.

When you push a version tag, GitHub Actions automatically builds:
- **macOS** — `PIIScrub-macOS.dmg` (drag-to-Applications installer)
- **Windows** — `PIIScrub-Windows-Setup.exe` (standard wizard installer)
- **Linux** — `PIIScrub-Linux.tar.gz` (extract-and-run tarball)

Builds take roughly **15–25 minutes** (most of that is downloading the 560 MB spaCy model on the CI runner — subsequent releases are faster due to caching).

---

## Step-by-step release

### 1. Bump the version

Edit two files:

**`src/piiscrub/__init__.py`**
```python
__version__ = "0.2.0"   # ← new version
```

**`pyproject.toml`**
```toml
version = "0.2.0"        # ← must match __init__.py
```

### 2. Update CHANGELOG.md

Add a section at the top:
```markdown
## [0.2.0] — 2024-XX-XX
### Added
- ...
### Fixed
- ...
```

### 3. Commit

```bash
git add src/piiscrub/__init__.py pyproject.toml CHANGELOG.md
git commit -m "Release v0.2.0"
```

### 4. Tag and push

```bash
git tag v0.2.0
git push
git push --tags
```

The `push --tags` line triggers the release workflow in `.github/workflows/release.yml`.

### 5. Monitor the build

Go to **Actions** tab on GitHub → select the `Release` workflow run.

If a build fails:
- Check the logs for the failing platform
- Fix the issue, delete the tag, re-tag:
  ```bash
  git tag -d v0.2.0
  git push origin :refs/tags/v0.2.0
  git tag v0.2.0
  git push --tags
  ```

### 6. Publish the release

The workflow creates the GitHub Release as a **draft** (so you can review before publishing).

1. Go to **Releases** on GitHub
2. Find the draft release for `v0.2.0`
3. Review the release notes (auto-generated by the workflow)
4. Click **Publish release**

Users can then download from the Releases page.

## First-time setup

The release workflow needs write access to create GitHub Releases. This is handled automatically via `permissions: contents: write` in the workflow file — no token setup required for public repos.

For **private repos**, create a Personal Access Token with `repo` scope and add it as a repository secret named `GH_TOKEN`, then update the workflow to use `${{ secrets.GH_TOKEN }}` instead of `${{ github.token }}`.

## Code signing (optional — eliminates OS security warnings)

Without code signing, users see:
- **macOS Gatekeeper**: "PIIScrub cannot be opened because it is from an unidentified developer"
  - Workaround: right-click → Open → Open, or run `xattr -cr /Applications/PIIScrub.app`
- **Windows SmartScreen**: "Windows protected your PC"
  - Workaround: click "More info" → "Run anyway"

To eliminate these warnings:

**macOS**: Requires an Apple Developer account ($99/year). Add `APPLE_CERTIFICATE`, `APPLE_CERTIFICATE_PASSWORD`, `APPLE_ID`, and `APPLE_TEAM_ID` secrets, then extend the workflow with `codesign` and `notarytool` steps.

**Windows**: Requires an EV (Extended Validation) code signing certificate (~$300/year). Add to the workflow with `signtool sign` before Inno Setup runs.

These are optional — SmartScreen/Gatekeeper warnings are a one-time click-through for users and don't affect functionality.

## Local test build

To verify packaging works before tagging a release:

```bash
# Install PyInstaller into the uv env
uv pip install pyinstaller

# Build (takes ~5 min on first run)
uv run pyinstaller piiscrub.spec --clean --noconfirm

# Test the binary
./dist/PIIScrub/PIIScrub
# → should print the startup URL and open your browser
```

On macOS, also test the DMG:
```bash
bash packaging/macos/create-app.sh
open PIIScrub-macOS.dmg
```
